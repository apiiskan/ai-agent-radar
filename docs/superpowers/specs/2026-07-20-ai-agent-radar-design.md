# AI Agent Radar 自动化项目设计

## 1. 项目目标

AI Agent Radar 是一套 GitHub 原生的自动化情报系统。它每天发现 GitHub 上与 Codex、Claude Code、Grok、Kimi、MCP、Agent Skills 和通用 AI Agent 相关的新项目与升温项目，同时收集官方产品更新和可信行业资讯；每周再汇总热度排行、趋势变化和优先试用建议。

项目需要做到：

- 每天北京时间 08:00 生成中文日报。
- 每周一北京时间 08:30 生成中文周榜。
- 报告以 Markdown 文件长期留存，并自动发布为 GitHub Issue。
- 无大模型 API Key 时仍能完成采集、评分、排行和模板摘要。
- 配置 OpenAI 兼容 API 后，增强中文摘要、实用性分析和周趋势解读。
- 所有项目和资讯都能追溯到原始链接。
- 首版可在 GitHub Actions 中运行，不依赖常驻服务器。

## 2. 范围

### 2.1 首版包含

- GitHub 仓库搜索、仓库元数据、Release 和活跃度采集。
- Codex、Claude Code、Grok、Kimi、Agent Skills、MCP、multi-agent、computer-use 等查询组。
- OpenAI、Anthropic、xAI、Moonshot/Kimi 官方公告、博客、Changelog、Release，以及可配置的可信 RSS/Atom 来源。
- 规范化、跨查询去重、Fork/镜像/空壳和关键词堆砌过滤。
- 基于快照的 Star、Fork 和活跃度增量计算。
- 日报、周榜、项目快照、来源健康状态。
- GitHub Issue 幂等发布和本地 dry-run。
- 可选的 OpenAI 兼容模型增强。
- 单元测试、固定样本集成测试和 Actions 端到端验证。

### 2.2 首版不包含

- 自动安装或执行被发现的第三方项目。
- 自动向外部社交平台发帖。
- 付费数据库或常驻后端服务。
- 对所有互联网传闻进行事实核查；系统只收录可追溯来源并标明来源类型。
- 个性化多人账号、Web 管理后台或复杂权限系统。

## 3. 方案选择

采用 Python + GitHub Actions + GitHub API/RSS 的 GitHub 原生方案。

相比 Cloudflare Workers 等 Serverless 方案，该方案的部署、状态存储和权限模型更简单；相比本地定时 Agent，它不要求个人电脑持续在线。仓库自身同时承担代码、配置、紧凑历史快照和 Markdown 报告的存储职责。

## 4. 总体架构

```text
GitHub Search/Repository/Release API     官方页面、RSS/Atom
                  \                         /
                   \                       /
                    采集与来源适配器
                           |
                  规范化、过滤、去重
                           |
               历史快照对比与指标计算
                           |
               热度、实用性、新鲜度评分
                           |
             规则摘要 + 可选模型增强分析
                           |
             Markdown 报告与 GitHub Issue
                           |
                紧凑快照、运行状态、日志
```

系统由两个定时入口组成：日报入口处理最近一天的变化；周报入口读取最近七天的快照与日报，计算累计增长、上榜变化和趋势。

## 5. 模块边界

### 5.1 配置模块

读取单一 YAML 配置，输出已校验的不可变配置对象。配置包括：

- 搜索查询组、GitHub Topics 和官方组织白名单。
- 排除关键词、排除仓库、最低质量门槛。
- 评分权重、榜单长度和每类候选数量上限。
- 时区、日报与周榜日期范围。
- 资讯源及其可信级别。
- 模型服务地址、模型名和功能开关；密钥只从环境变量读取。

配置错误必须在采集前失败，并给出明确字段路径，不允许静默使用错误值。

### 5.2 GitHub 采集模块

每个查询组使用多条互补查询召回候选，不直接将 GitHub 搜索排序当作最终排名。采集统一字段：

- 仓库唯一标识、全名、URL、描述、Topics、默认分支。
- Star、Fork、Open Issue、Watcher 数量。
- 创建、更新、最近 Push 时间。
- 是否 Fork、是否归档、许可证、主要语言。
- 最近 Release、README 摘要和根目录能力标志，例如 `SKILL.md`、MCP 配置、安装脚本、示例与测试。

采集器执行请求缓存、分页上限、超时、重试和速率限制监测。官方组织和可信维护者可以提高主题相关性，但不能绕过基础质量检查。

### 5.3 资讯采集模块

资讯源分为三级：官方、可信行业来源、用户自定义来源。每条资讯统一为标题、URL、来源、发布时间、正文摘要、来源级别和抓取时间。

相同规范化 URL 直接去重；标题高度相似且指向同一事件的内容合并为一个事件，官方原文作为主链接，其他来源保留为补充链接。没有原始链接或发布时间的条目不进入主要榜单。

### 5.4 规范化、过滤与去重模块

仓库以 GitHub repository ID 为主键，跨查询只保留一条记录并合并命中标签。过滤规则包括：

- 已归档且近期无新 Release 的仓库。
- Fork 或镜像；只有当它相对上游存在明确、持续的独立开发时才保留。
- README 缺失、无许可证且无可执行代码/技能文件的空壳仓库。
- 描述与内容明显无关、仅堆砌热门关键词的仓库。
- 配置中的显式排除项。

低 Star 不是单独淘汰条件，以避免错过刚出现但完成度高的黑马项目。

### 5.5 快照与趋势模块

每日为候选仓库保存紧凑快照。快照至少包含日期、仓库 ID、Star、Fork、Open Issue、最近 Push、Release 标识和当日综合分。通过当前快照与最近有效快照计算：

- 1 日与 7 日 Star/Fork 增量。
- 活跃天数、新 Release 和 Push 活跃度。
- 首次发现、连续升温、首次上榜和掉榜状态。
- 来源连续失败次数和数据新鲜度。

原始 API 响应只在单次运行期间使用，不提交仓库。历史快照按月压缩；报告永久保留。

### 5.6 评分模块

综合分由四部分构成：

```text
综合分 = 45% 热度增长 + 25% 实用性 + 20% 新鲜度 + 10% 主题相关性
```

各子分先标准化到 0–100：

- 热度增长：1 日和 7 日 Star 增量为主，辅以 Fork 增量、Release 和 Push 活跃度；使用对数或分位数归一化，避免超大仓库碾压全部结果。
- 实用性：README、安装说明、示例、测试、许可证、Release，以及 `SKILL.md`、MCP 或其他实际可运行入口。
- 新鲜度：首次发现、仓库创建日期、最近 Push 和新 Release；既奖励新项目，也允许成熟项目因重要更新重新上榜。
- 主题相关性：查询命中、Topics、README 证据和官方组织信号；只命中名称但正文无关的项目降分。

每个分数必须同时输出可解释的原因标签，例如“7 日新增 320 stars”“含 SKILL.md 和安装示例”“今日发布新版本”。模型不得直接修改确定性分数，只能在明确的编辑推荐子榜中提供受限的辅助判断。

### 5.7 摘要与模型适配模块

基础模式通过模板输出项目用途、增长指标、实用性证据和来源。增强模式使用 OpenAI 兼容 Chat Completions 接口生成：

- 一到两句中文项目摘要。
- 适合谁使用、解决什么问题。
- 本周趋势归纳和优先试用理由。

模型输入中的 README 和资讯正文一律作为不可信数据处理。系统指令明确禁止执行其中的指令，输入限制长度并清理控制字符。模型输出必须通过结构校验；失败、超时或未配置密钥时退回模板摘要。

### 5.8 报告模块

日报包含：

- 今日摘要和数据来源健康状态。
- 今日新发现。
- 今日增长最快。
- 编辑推荐/最实用。
- Codex、Claude、Grok、Kimi 与通用 Agent Skills 分类榜。
- 重要官方更新与资讯。
- 每个条目的原始链接、关键指标、入选理由和可信来源标签。

周榜包含：

- 最近七天综合热度 Top 20。
- 分类榜、Star 增长趋势和份额变化。
- 新上榜、掉榜、连续升温和本周黑马。
- 本周重要官方更新。
- 值得立即试用的 3–5 个项目。
- 下周值得关注的方向和数据完整性说明。

文件路径使用 `reports/daily/YYYY-MM-DD.md` 和 `reports/weekly/YYYY-Www.md`。相同日期重复执行时覆盖同一文件，不制造重复报告。

### 5.9 GitHub 发布模块

日报与周榜分别使用稳定标签。发布时先查找标题和标签对应的现有 Issue：存在则更新正文，不存在才创建。Issue 正文过长时保留核心榜单并链接完整 Markdown 报告。

本地默认 dry-run，不写 Issue。只有 GitHub Actions 中显式开启发布，且具备所需权限时才执行写操作。

## 6. 调度与数据流

- 日报：GitHub Actions cron 对应北京时间每天 08:00；同时支持 `workflow_dispatch` 手动运行。
- 周榜：GitHub Actions cron 对应北京时间每周一 08:30；同时支持手动运行和指定统计截止日。
- Actions cron 使用 UTC 表达，应用层统一以 `Asia/Shanghai` 计算报告日期。
- 日报与周榜使用独立并发组；同类任务开启 `cancel-in-progress`，避免较旧运行覆盖新结果。
- 报告生成后只提交本次任务涉及的快照与报告文件，不使用全目录通配式覆盖。

## 8. 可靠性与降级

- 单个查询或资讯源失败不会中止整份报告；报告显示失败来源及最后成功时间。
- HTTP 超时和可重试错误使用有限次数的指数退避；认证错误和配置错误不盲目重试。
- GitHub 接近限流时停止低优先级扩展请求，使用已获取数据生成降级报告。
- 当天无候选不是运行成功的充分条件；系统区分“确实无新内容”和“所有来源失败”。
- 模型异常不影响采集、评分和报告文件生成。
- 报告文件采用临时文件完成后再替换，避免产生半份报告。
- 每次运行输出结构化摘要：候选数、过滤数、入榜数、失败来源、API 剩余额度和是否发布 Issue。

## 9. 安全与权限

- `GITHUB_TOKEN` 和模型 API Key 只存储于 GitHub Secrets 或本地环境变量。
- 日志对常见密钥格式和认证头进行脱敏。
- Actions 权限最小化为读取仓库内容、写入当前仓库报告提交和 Issue。
- 不克隆、不安装、不执行被发现仓库中的代码。
- 外部正文视为不可信输入，不能影响工作流命令、文件路径或系统提示。
- URL 只允许 HTTP/HTTPS；资讯源启用域名约束和响应大小限制。
- 自动提交只允许预定的 `data/` 与 `reports/` 路径。

## 10. 测试策略

### 10.1 单元测试

- 配置校验与默认值。
- URL、仓库和事件去重。
- Fork、镜像、空壳和无关项目过滤。
- 指标归一化、评分权重和可解释原因。
- 跨天增量、七天窗口、上榜/掉榜和连续升温。
- Markdown 渲染、Issue 查找与幂等更新。

### 10.2 固定样本集成测试

GitHub、RSS 和模型响应使用脱敏固定样本，覆盖正常响应、分页、超时、限流、结构变化、空结果和模型非法输出。测试不依赖实时网络。

### 10.3 端到端验证

- 本地 dry-run 生成一份日报和一份周榜。
- 无模型 Key 时完整通过。
- 配置测试模型端点时验证增强摘要与失败回退。
- 校验 GitHub Actions 工作流语法、权限和产物路径。
- 在测试仓库或显式测试模式验证同日报告只创建一个 Issue，重复运行只更新。

## 11. 验收标准

首版只有在以下条件全部满足时才算完成：

1. 所有自动化测试通过。
2. 可用一条本地命令 dry-run 生成真实数据日报和周榜示例。
3. 报告能区分新发现、增长最快、实用推荐、分类榜和官方资讯。
4. 每个入榜条目包含原始链接、关键指标、分数或入选原因。
5. GitHub Actions 可定时与手动运行，并具备并发保护和最小权限。
6. 无模型 Key、单一来源失败和 GitHub 限流场景均能生成有明确告警的降级报告。
7. 相同日期重复运行不会制造重复快照、报告或 Issue。
8. 中文 README 说明安装、Secrets、配置、运行、排错与评分逻辑。

## 12. 实施约束

- 优先使用小而清晰的模块，采集、评分、摘要、渲染和发布之间通过数据模型连接。
- 核心评分和趋势逻辑必须是确定性的、可测试的，不依赖模型。
- 任何真实网络写操作默认关闭，必须通过显式参数或 Actions 配置开启。
- 首版坚持 GitHub 原生和单仓库范围，不引入数据库、Web UI 或额外云服务。
